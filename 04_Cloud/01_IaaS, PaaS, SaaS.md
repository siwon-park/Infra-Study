## 00_On Premise

`고객이 직접 서버를 구축하고 운영`하고 책임지는 방식을 말하며, Packaged Software라고도 한다

### 요약

- 물리적인 장치, 하드웨어(CPU, RAM, Storage, Network device 등등)을 모두 직접 구매
- 직접 OS를 설치
- 네트워크 환경을 직접적으로 구성
- 서버 관리를 직접적으로 해야 함(트래픽, 프로지버닝 등등)
- 모든 것을 직접 사용자가 다 준비해야 하기 때문에 많은 시간과 돈을 소비

<br>

## 01_IaaS (Infrastructure as as Service, 인프라 기반 서비스)

서버를 운영하기 위한 `IT 인프라를 서비스로 제공`하는 형태로,  `서버, 스토리지 및 네트워크 등과 같은 사용자가 필요한 컴퓨팅 자원(IT 인프라)을 가상화된 형태로 제공하는 서비스`를 말한다.

> 극장 주인로부터 극장을 빌려서, 무대를 스스로 만들고 공연까지 준비하는 방법
>
> - 극장 주인: 클라우드 제공 업체
> - 극장 및 무대: 인프라
> - 공연: 어플리케이션

### 장점

#### 필요한 만큼의 자원을 동적으로 할당 가능

- 기업 규모에 따라 유연하게 인프라를 확장 가능하고, 당양한 운영체제, 언어, 프레임워크 등을 지원함
- 기존 On-premise 방식보다 확장성이 좋음

#### 비용을 더욱 효율적으로 관리 가능

- 사용자는 IT 인프라를 직접 구축할 필요 없이 클라우드 제공업체가 제공하는 가상의 인프라를 활용해서 OS를 직접 올리고 그 상위 계층만 구성하면 되므로 `초기 비용 유지 및 유지보수 비용 등을 절감할 수 있음`
- 요청에 따라 자원 사용량을 조정할 수 있고, 사용한 자원만큼 비용을 지불하므로 비용 절감 가능

#### 인프라 확장이 빠르고 용이함

- 기업이 점진적으로 자신의 IT 인프라를 확장할 수 있도록 지원하여, 급격한 규모의 변화에 대응 가능
- 고객은 가상 서버 하위의 레벨에 대해서는 고려할 필요 없음

#### 보안성을 높일 수 있음

- 제공받는 인프라에 대한 보안성을 가진 업체로 부터 보안 기능을 제공 받아 안정성을 높일 수 있음

#### 필요한 기능을 빠르게 개발 가능

- 서비스 제공 업체에서 미리 준비한 템플릿을 활용해 빠르게 어플리케이션을 개발하고 배포 가능

<br>

### 단점 및 도입 고려 사항

#### 기술적인 지식 필요 및 운영의 부담

- 사용자가 직접 인프라를 관리해야 하므로, 개발자 외에 인프라 관리 및 유지보수를 할 수 있는 인력이 추가적으로 필요함

#### 서비스 제공업체에 의존적

- 서비스 제공업체에 장애가 발생할 경우, 사용자의 서비스에도 영향을 미침
- 제공되는 인프라의 제한으로, 사용자가 원하는 모든 기능을 제공할 수 없을 수도 있음

#### 네트워크를 통한 서비스 제공으로 성능적 문제 발생 가능성

- 대규모 데이터 전송이나 복잡한 애플리케이션을 실행하는 경우나 다른 사용자들의 활동에 따라 성능 문제가 발생할 수 있습니다.

<br>

### 예시

Amazon Web Service EC2(AWS EC2)

- AWS의 EC2를 이용하면 우리는 물리적인 서버와 Network, Storage 등등을 직접 구매하거나 준비하지 않아도 원하는 OS를 깔아 서버로 사용 가능
- AWS의 EC2는 사용자가 원하는 OS를 고르고 그에 해당하는 스펙을 선택하기만 하면, 모든 관리를 아마존에서 해줌. 
- OS를 제공한다는 느낌이긴 하지만, 선택권을 주고 OS의 종류나 다양한 자원들을 사용자가 선택하므로 대표적인 IasS라고 불리고 있음

<br>

## 02_PaaS (Platform as a Service, 플랫폼 기반 서비스)

`고객(개발자)이 응용 프로그램(어플리케이션)을 개발, 실행, 테스트, 관리 등을 할 수 있도록 플랫폼 및 환경을 서비스로 제공`하는 모델

> 극장 주인에게 극장을 빌리고, 내가 원하는 무대까지 만들어 달라고 요청하고 공연만 스스로 준비하는 방법

- `IaaS`와 헷갈릴 수 있는데 아마존과 같은 서비스가 VM을 제공하는 것이 IaaS라면, `PaaS`는 `node.js`, `Java`와 같은 런타임을 미리 깔아놓고, 거기에 소스코드를 넣어서 돌리는 구조 => 우리는 소스코드만 적어서 빌드 하는 것이고, 컴파일은 클라우드에서 하여 결과만 가져오는 거라고 생각하면 됨

### 장점

#### 인프라 및 관리 비용을 절감 가능

- 서버, 스토리지, 네트워크 등의 하드웨어 인프라를 구매할 필요가 없어 비용 절감 가능
- 어플리케이션 운영 및 관리 자동화를 통한 인력 비용 절감 가능
- `IaaS` 대비 비용 저렴
  - 필요한 플랫폼만 소비 형태로 사용하기 때문에 비용 부담이 상대적으로 낮음

#### 개발 생산성의 증가

- 개발자가 어플리케이션 코드 개발에만 집중할 수 있도록 개발에 필요한 미들웨어, DB, 도구 등을 지원
- 개발 및 배포 프로세스를 빠르게 확보 가능
  - 이미 설치된 미들웨어 위에 코드만 돌리면 되기 때문에, 관리가 매우 편리함 => 개발자들은 Infra 영역의 수고를 덜어 오직 개발에만 집중 가능

#### 어플리케이션의 확장성 용이

- 필요에 따라 인프라를 수평/수직적으로 확장 가능하며 동적으로 자원을 할당하거나 해제 가능
- 탄력적 리소스 관리
  - `IaaS`가 제공하는 이점 또한 가지기 때문에 만약 개발한 애플리케이션의 사용자가 급증하여서 리소스가 추가로 필요해지면 이를 자동으로 늘리고, 다시 사용자가 감소하면 다시 그의 맞는 리소스로 되돌려서 비용을 절감 가능

#### 어플리케이션 운영 자동화 및 안정성, 보안성 보장

- 백업, 복구, 보안 등의 작업을 자동으로 처리하여 어플리케이션의 안정성 유지 가능

<br>

### 단점

#### 제한된 제어 수준

- 개발자가 인프라에 대한 세밀한 정보, 제어하는 행동은 제한될 수 있음

#### 보안상 문제

- 공급업체의 보안 정책 및 수준에 따라 민감한 데이터를 저장함에 있어 개발자가 지속적으로 신경써야함

#### 공급업체에 의한 높은 의존도

- IaaS보다 공급업체가 제공하는 서비스에 대한 의존도가 높아져서, 개발자가 원하는 도구와 서비스를 선택할 수 있는 유연성이 제한됨
- 개발한 어플리케이션이 플랫폼에 종속되어 개발되기 때문에 다른 플랫폼으로의 이동이 어려울 수도 있음(특정 플랫폼 종속)

#### 제한된 커스터마이징

- 개발자가 특정 요구 사항에 맞게 인프라를 커스터마이징 하는 것이 제한됨

#### 비용 예측의 어려움

- 비용 효율적일 수는 있으나, 사용량이 예기치 않게 증가하거나 수요가 급증하면 비용 예측이 어려워 개발자가 지속적으로 비용을 모니터링해야 함

<br>

### 예시

Heroku, Google App Engine 등

<br>

### 언제 사용하는 것이 적절?

신속한 개발을 원할 경우

- 동일한 개발 프로젝트를 수행하는 여러 개발자가 있거나 다른 공급 업체도 포함해야하는 경우 PaaS는 전체 프로세스에 뛰어난 속도와 유연성을 제공
- 사용자 정의 응용 프로그램을 만들려는 경우에도 유용
- 앱을 신속하게 개발하거나 배포하는 경우에 발생하는 몇 가지 문제를 단순화 가능

<br>

## 03_SaaS (Software as a Service, 소프트웨어 기반 서비스)

`공급 업체가 클라우드를 통해 모든 기능을 서비스로 제공하며 관리`해주는 모델

공급 업체가 서버 구성, 보안 및 업그레이드와 같은 기술적인 부분을 전면 책임지고, 고객은 소프트웨어를 사용하면서 라이선스 비용을 지블하고 필요한 만큼 서비스를 이용 가능

### 장점

#### 비용의 효율적 사용

- 소프트웨어의 유지보수나 업그레이드에 대한 추가 비용이 없으며, 제공자가 책정한 요금에 맞게 필요한 만큼 비용을 지불하여 서비스를 운영 가능함

#### 소프트웨어의 설치 및 유지보수 용이

- `SaaS`의 경우 대부분 웹 인터페이스를 사용하여 소프트웨어의 설치 및 유지보수가 쉬움
- 사용자는 기술적인 문제를 해결하는 것보다 핵심 업무에 집중할 수 있음

#### 필요에 따른 확장 용이

- 고객의 필요에 따라 추가 기능을 구매하거나, 사용자 수를 늘리거나, 서버 용량을 늘리거나, 지역적으로 확장 가능

#### 보안적인 기능에 대한 걱정 無

- `SaaS` 제공 업체는 데이터 보안과 백업, 복원과 같은 보안 문제를 처리할 수 있기 때문에 고객은 보안 문제에 대해 걱정할 필요가 없으며, 데이터 손실 및 손상에 대한 위험을 줄일 수 있음

### 단점 및 고려 사항

#### 네트워크로부터 많은 영향을 받음

- 클라우드 서비스를 기반으로 제공되기 때문에 반드시 인터넷이 있어야 하며, 인터넷 연결이 끊어지면 소프트웨어를 사용할 수 없음
- 인터넷 연결 속도에 영향을 받기 때문에 로컬에서 실행되는 속도보다는 실행 속도가 느림

#### 데이터 보안에 유의

- 데이터가 클라우드 서버에 저장되기 때문에 정보 유출의 위험성이 존재함

#### 서비스 제공업체에 대한 의존성

- `SaaS` 솔루션은 일반적으로 일반적인 요구사항에 맞춰져 있기 때문에, 특정 기업의 요구에 맞게 맞춤화하는 데는 한계가 있음 => 일부 사용자에게는 제한적인 기능을 제공할 수도 있음(커스터마이징 제약)

<br>

### 언제 사용하는 것이 적절?

비즈니스에 집중하고 싶은 경우

- 보안상 민감한 사항이 아니라면 모든 기업에게 SaaS는 훌륭한 옵션
- 협업이 필요한 단기 프로젝트라면 SaaS를 도입하는 것이 훨씬 유리
- 일반적으로 On-premise 솔루션은 모바일 액세스를 지원하지 않기 때문에 모바일 액세스가 필요한 경우에도 SaaS를 사용하면 비용과 시간을 절약 가능

<br>

### 예시

구글 클라우드, MS오피스 365

<br>

## 4. FaaS (Function as a Service, 서비스형 기능)

함수나 API 등 개발한 하나의 모듈적인 기능을 서비스로서 제공하는 것을 말한다.

쉽게 말해 "함수를 서비스로 제공한다"고 이해하면 된다.

예를 들면, AWS Lambda에 개발한 함수나 기능의 소스를 올리면 해당 함수나 기능을 서비스로서 제공이 가능하다.

흔히 FaaS를 이야기하면 `서버리스(serverless, 서버가 없음)`이라는 말이 붙어 다닌다. 그러나 사실 엄밀히 말하면 서버가 없을 수는 없다. 서버가 없는데 어떻게, 무엇을 통해서 기능이나 함수를 제공한다는 말인가?

단지 서버를 따로 관리하지 않아도 되는 것을 서버리스라고 한다.

즉, FaaS는 서버리스 아키텍처(Serverless Architecture) 중 하나이다.
